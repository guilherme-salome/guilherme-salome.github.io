Emacs Running Python on Remote Server

February 25, 2025
Emacs Org-Mode Babel Python

Create an executable that will run the code necessary to start Python in the remote server. My script is named remote-python:

#!/usr/bin/env bash
ssh hpclogin -t "module load python/3.12; python $@"
# You could also change the working directory and load a virtual environment:
# ssh hpclogin -t "module load python/3.12; cd ~/Guilherme/CRON/aims-test/; source pyenv/bin/activate; python $@"

Make the script executable:

chmod +x remote-python

Modify how Emacs launches the Python interpreter:

(setq python-shell-interpreter "/Users/C304095/Library/CloudStorage/GoogleDrive-guilhermesalome@gmail.com/My Drive/Emacs/bin/remote-python"
      python-shell-interpreter-args "-i"
      python-shell-completion-native-enable nil)

Starting the inferior python buffer with M-x run-python works and you can send Python code to that buffer with M-x python-shell-send-buffer. However, I could not get the Babel code blocks to properly execute in the remote server.

print("Hello!")

Evaluating the code block above fails with a FileNotFoundError because Babel seems to be trying to write the code to a temporary file, and then getting it evaluated in the session, instead of sending the code directly as text.

The function that is called when the code block is executed is org-babel-execute:python:

(defun org-babel-execute:python (body params)
  "Execute Python BODY according to PARAMS. This function is called by `org-babel-execute-src-block'."
  (let* ((org-babel-python-command
	  (or (cdr (assq :python params))
	      org-babel-python-command))
	 (session (org-babel-python-initiate-session
		   (cdr (assq :session params))))
	 (graphics-file (and (member "graphics" (assq :result-params params))
			     (org-babel-graphical-output-file params)))
         (result-params (cdr (assq :result-params params)))
         (result-type (cdr (assq :result-type params)))
	 (return-val (when (eq result-type 'value)
		       (cdr (assq :return params))))
	 (preamble (cdr (assq :preamble params)))
	 (async (org-babel-comint-use-async params))
         (full-body
	  (concat
	   (org-babel-expand-body:generic
	    body params
	    (org-babel-variable-assignments:python params))
	   (when return-val
	     (format (if session "\n%s" "\nreturn %s") return-val))))
         (result (org-babel-python-evaluate
		  session full-body result-type
		  result-params preamble async graphics-file)))
    (org-babel-reassemble-table
     result
     (org-babel-pick-name (cdr (assq :colname-names params))
			  (cdr (assq :colnames params)))
     (org-babel-pick-name (cdr (assq :rowname-names params))
			  (cdr (assq :rownames params))))))

We will try to overwrite this function so that we send the code directly to the Python buffer:

(defun my-org-babel-execute:python (body params)
  ;;
  (message (concat "This is the body:\n" body))
  (let ((proc (python-shell-get-process))) (python-shell-send-string body proc) "")
  )

(advice-add 'org-babel-execute:python :override #'my-org-babel-execute:python)

We can now start the Python inferior process with M-x run-python, and then execute a code block, and it will get sent to the associated buffer.

The function python-shell-send-string is behind this issue, since the message prints the content of the code block correctly. Inspecting the send string function:

(defun python-shell-send-string (string &optional process msg)
  "Send STRING to inferior Python PROCESS.
When optional argument MSG is non-nil, forces display of a
user-friendly message if there's no process running; defaults to
t when called interactively."
  (interactive
   (list (read-string "Python command: ") nil t))
  (let ((process (or process (python-shell-get-process-or-error msg)))
        (code (format "__PYTHON_EL_eval(%s, %s)\n"
                      (python-shell--encode-string string)
                      (python-shell--encode-string (or (buffer-file-name)
                                                       "<string>")))))
    (unless python-shell-output-filter-in-progress
      (with-current-buffer (process-buffer process)
        (save-excursion
          (goto-char (process-mark process))
          (insert-before-markers "\n"))))
    (if (or (null (process-tty-name process))
            (<= (string-bytes code)
                (or (bound-and-true-p comint-max-line-length)
                    1024))) ;; For Emacs < 28
        (comint-send-string process code)
      (let* ((temp-file-name (with-current-buffer (process-buffer process)
                               (python-shell--save-temp-file string)))
             (file-name (or (buffer-file-name) temp-file-name)))

There is a comint-max-line-length that seems to trigger the writing to a temporary file. I thought about using python-shell-send-buffer, but it also ends up calling the same python-shell-send-string function.

Another function we could use to send the code is comint-send-string.

(defun my-org-babel-execute:python (body params)
  ;;
  (message (concat "This is the body:\n" body))
  (let ((proc (python-shell-get-process))) (comint-send-string proc (concat body "\n")) "")
)

(advice-add 'org-babel-execute:python :override #'my-org-babel-execute:python)

This function seems to solve the problem and even the larger code block is sent to the inferior buffer.

Increasing the comint-max-line-lenght seems to fix the problem, but also requires going back to python-shell-send-string to work:

(defun my-org-babel-execute:python (body params)
  ;;
  (message (concat "This is the body:\n" body))
  (setq comint-max-line-length 100000)
  (let ((proc (python-shell-get-process))) (python-shell-send-string (concat body "\n") proc) "")
)

(advice-add 'org-babel-execute:python :override #'my-org-babel-execute:python)
